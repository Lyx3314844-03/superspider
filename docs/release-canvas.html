GitHub Release Canvas

One crawler brand. Four very different runtime choices.

SuperSpider is published as a clean four-runtime release surface: PySpider for fast AI-heavy project authoring, GoSpider for concurrent binary deployment, RustSpider for strongly typed high-performance releases, and JavaSpider for browser workflow and enterprise Java delivery.

PySpider GoSpider RustSpider JavaSpider

Release focus

This release strips the repository down to what matters in public: the four shipped runtimes, the three operating-system install versions, and the capability documents needed to choose between them.

  • Four runtime directories only
  • Windows, Linux, and macOS installers for each runtime
  • Release-facing docs instead of monorepo sprawl
  • Capability-first presentation for framework selection

Four frameworks, four strengths

These runtimes solve the same class of crawling problems, but each one is shaped around a different production reality.

PySpider

fastest iteration loop

PySpider is the strongest choice when the crawl is a living project: evolving schemas, AI extraction, plugin injection, browser + HTTP mixing, and data workflows all in one place.

  • Python-native CLI and package workflow
  • Richest scrapy-style project runtime
  • AI extraction, research, and plugin surfaces
  • Browser + HTTP hybrid crawling
  • Media, anti-bot, dataset, and node-reverse support

GoSpider

binary-first concurrency

GoSpider fits teams that want a service-friendly executable, strong concurrency, and a runtime that is straightforward to deploy across workers, queues, and operational nodes.

  • Compiled CLI with simple deployment shape
  • Concurrent crawling and task dispatch
  • Distributed workers, queues, and storage
  • Browser artifact capture and replay
  • Media download, anti-bot, and control-plane-friendly runtime

RustSpider

typed release boundary

RustSpider is built for high-performance release binaries, stricter runtime boundaries, and teams that want to decide exactly which modules ship by using feature gates.

  • Rust release binary output
  • Feature-gated browser, distributed, API, and web modules
  • Typed scrapy-style interfaces
  • Preflight, monitoring, and contract-heavy runtime posture
  • Strong fit for performance-sensitive deployments

JavaSpider

browser workflow + Maven

JavaSpider fits organizations already living in Java build pipelines and browser-heavy automation, where Maven packaging, auditability, and enterprise workflow integration matter.

  • Maven / JAR packaging model
  • Browser workflow emphasis with Selenium and Playwright helpers
  • Audit, connector, and session-aware execution paths
  • Scrapy-style compatibility surface
  • Workflow replay, media parsing, and enterprise integration

Three operating-system install versions

Every runtime ships with its own install path for Windows, Linux, and macOS, so users choose the framework first and the operating system second.

Windows

scripts/windows/install-pyspider.bat scripts/windows/install-gospider.bat scripts/windows/install-rustspider.bat scripts/windows/install-javaspider.bat

Linux

scripts/linux/install-pyspider.sh scripts/linux/install-gospider.sh scripts/linux/install-rustspider.sh scripts/linux/install-javaspider.sh

macOS

scripts/macos/install-pyspider.sh scripts/macos/install-gospider.sh scripts/macos/install-rustspider.sh scripts/macos/install-javaspider.sh

Capability range

This release is designed so each runtime has a clear delivery boundary, not just a language label.

PySpider

Best when the crawler is an evolving project rather than a single packaged binary.

  • Project runtime and plugin injection
  • AI extraction and research workflows
  • Hybrid browser + HTTP crawling
  • Flexible Python-side data handling

GoSpider

Best when the crawler is an executable unit that needs concurrency and operations-friendly rollout.

  • Binary-first deployment
  • Concurrent task dispatch
  • Worker / queue / storage model
  • Service-side crawling workloads

RustSpider

Best when the release boundary must be explicit and performance is a first-order concern.

  • Strongly typed release binary
  • Feature-gated shipping surface
  • Preflight and contract-heavy runtime
  • High-performance deployment profile

JavaSpider

Best when browser workflows must sit inside Maven packaging and Java enterprise pipelines.

  • Maven / JAR delivery
  • Browser workflow execution
  • Audit and connector paths
  • Enterprise Java integration

What the repository now contains

The default branch is intentionally narrow. It keeps only the four crawler runtimes, the install scripts, and the capability docs so GitHub opens as a product release surface instead of an experimental crawler monorepo.

  • Source: `pyspider`, `gospider`, `rustspider`, `javaspider`
  • Install docs: `docs/SUPERSPIDER_INSTALLS.md`
  • Primary docs: `docs/DOCS_INDEX.md`, `MEDIA_PARITY_REPORT.md`, `LATEST_FRAMEWORK_COMPLETION_REPORT.md`, `docs/archive/HISTORICAL_REPORTS.md`
  • Brand: `docs/assets/superspider-wordmark.svg`, `docs/assets/superspider-icon.svg`
